# 完成形：AI協働における「構造」の必要性

2026-07-05。非規範。各論の入口。

---

AI にコードを書かせること自体は、もはや驚きでも技術的なマジックでもない。現代の開発において真に驚くべきは、その凄まじい生成速度のあとに、プロジェクトとして「何も残らない」ことが頻発することだ。残るのは散逸したチャットの断片と、いきなり数百行増えた diff だけ。誰がどのタイミングで何を承認したのか、なぜ他の選択肢ではなくその設計になったのか、次の担当者がシステムのどこから読み解けばよいのか——ソフトウェア開発において最も価値のあるこれらの文脈が、すべて霧散してしまう。私は、そのような再現性のない生成作業を「ソフトウェア・エンジニアリング」とは呼ばない。

コーディングエージェントはサンドボックスのなかで驚くほど自由に動く。だが、個々の要素が自由に動くことと、それらが統合されて一つのプロジェクトになることは全く別の次元の問題である。AI と人間が長期的に協調するためには、Operating Path（運用パス）、操作契約、Adjudicator（人間の裁定者）の承認ループ、そして永続化される成果物という**チーム側の強固な構造的仕組み**が不可欠である。この根本的な前提は、[エージェントはサンドボックスである](2026-07-07-rationale-saas-agent-as-sandbox.md) でさらに深く展開する。

このテンプレートがめざすのは、AI が人間の代わりにすべてを終わらせてくれる魔法の箱を作ることではない。AI と人間が**協調して**開発を持続的に進めるための、頑健な「足場（Scaffolding）」を提供することである。アドホックな会話を、再現性とレビュー可能性を備えた協働プロセスへと移す。これは、開発速度を落として後から無理やり統制（ガバナンス）を被せるということではない。Adjudicator・Agent・Deterministic Tool という三者の役割と、契約・成果物・承認点をシステムの前提として先に置くということだ。統制は速度に対するペナルティではなく、高速な生成を持続可能なプロダクトへ変換するための構造そのものである。

## 完成形の中心：三つの問い

本テンプレートが目指す「完成形」とは、AI が自律的にすべてを判断し、人間が眠っている間にプロジェクトが終わっているような状態ではない。むしろ逆である。AI が自由に設計空間を探索し、大量のコードを生成し、機械的な検証を補助できる一方で、プロジェクトの「意味」や「責任」が変わる特異点においては、人間が確実にプロセスを止め、意思決定を下せる状態こそが完成形である。責任を AI に丸投げするのではなく、AI の圧倒的な推論・生成能力を、人間のアーキテクチャ判断へと安全に接続するのだ。

この中心には、すべての変更（Pull Request や Diff）に対して問われるべき三つの根源的な問いがある。

1. **その変更は、ビジネスやドメインのどのような要求を満たすのか。**（What & Why）
2. **その判断は、プロジェクトのどこに記録されたのか。**（Where is the Truth）
3. **次にこのコードを触る人間（またはエージェント）は、どこから文脈を再開できるのか。**（Resumability）

この三問に明確に答えられない diff は、たとえ単体テストがすべて Green であっても、ソフトウェア工学的には「未完成」である。完成とは、コードがコンピュータ上で動作することだけでなく、人間がその来歴を読み取り、責任の所在を追跡できる状態を含んでいる。

## 契約のもとで働くということ

人間の設計者（Adjudicator）と AI エージェントは、明示的に書かれた運用契約のもとでだけ協働すべきである。承認と裁定は人間が担い、生成と機械的検証はエージェントが担う。この責任分担は、その日のエンジニアの気分やチャットプロンプトの揺らぎによって変わるものではなく、`AGENTS.md` や `CLAUDE.md` というリポジトリ内のコードとして強固に固定される。README はこれを "a shared, written operating contract"（共有され、文書化された運用契約）と呼んでいるが、要するに「AIとの関わり方を口約束や個人の暗黙知で済ませない」という強い宣言である。

ソフトウェア開発の歴史において、ソースコードそのものよりも先に、チームのプロセスや作法が壊れることは珍しくない。レビューを飛ばす。Issue のステータスを更新しない。テストを実装の後に辻褄合わせで作る。CIのスクリプトを手元でこっそり書き換える。AI はこの「プロセスの逸脱」を、人間の手作業とは比較にならない速度で実行・増幅する。だからこそ操作契約は、エージェントへの単なる命令書であると同時に、人間が自分たち自身の協働の規律を保つための鏡でもある。

## 判断は消えてはならない：Docs-as-Code

仕様、アーキテクチャの決定、計画、作業記録、そして引き継ぎ（handoff）——ソフトウェア開発におけるすべてのコンテキストは、リポジトリ内のバージョン管理された成果物でなければならない。EARS/Gherkin フォーマットによる受入仕様、ADR（アーキテクチャ決定記録）、local issue と work plan、trace、そして handoff 文書。SaaS のチャットログは、検索不能で揮発性の高いノイズであり、成果物ではない。将来の保守者が変更の来歴を論理的にたどれることは、システムの単なる「おまけ」ではなく、プロダクトの品質を構成する不可欠な一部である。

プロセスをコードと同様にリポジトリで管理する Docs-as-Code の哲学（[Write the Docs](https://www.writethedocs.org/guide/docs-as-code/)）の延長として、このテンプレートは「エージェントの挙動を定義する契約ファイル」までを版管理と CI の対象にする。ドキュメントにレビューと CI を適用するのがモダンな開発の常識であるなら、エージェントの出力挙動を左右する文書（プロンプト）に同じ規律を適用するのは当然の帰結である。

Michael Nygard による ADR 提言（[Cognitect](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions)）は、重要な決定を短いテキストとして版管理下に残すという極めて実務的なパラダイムを示した。本テンプレートはこの思想を、システムアーキテクチャの決定だけでなく、「AI とどのように協働したか」という運用決定の領域へまで拡張している。

## テンプレートはドメインを決めない：境界の謙虚さ

技術スタック、データストアの選定、外部プロバイダ、そしてコアとなるドメインロジック——本テンプレートは、これらプロジェクト固有の領域には一切関与しない。提供するのは、あくまでプロセスと協働のための「足場（Scaffolding）」だけである。未決のアーキテクチャは、エージェントが推測して適当に埋めてよい空白ではなく、`Current Non-Decisions` として明示的に「今は未決である」とマークされる。エージェントに対して「よしなにやっておいて」とは絶対に言わない。「未決の事項があるなら、人間の Adjudicator に ADR の作成を要求せよ」と命じるのだ。

これはテンプレートとしての「謙虚さ」の表れである。運用テンプレートは、採用先プロジェクトのビジネスドメインや技術的制約を知っているふりをしてはならない。協働のプロセスは提供できる。しかし、どの datastore が最適か、どの provider を信頼すべきか、どの業務概念を entity と呼ぶかは、完全に採用先のコンテキストと Adjudicator の設計判断に属する。この境界線を越えた瞬間、テンプレートは開発の支援ではなく、プロジェクトへの侵略となる。

## 完成形は静的ではない：Pull型同期と自己進化

テンプレートは、プロジェクトにコピー（clone）された瞬間から陳腐化が始まる。だからこそ、上流のアップデートは強制的な push ではなく、プロジェクト側からの主体的な pull で同期され、下流（プロジェクト側）の主権を 3-way merge によって保護する（ADR 0008）。プロジェクト特有の改善は下流から還流し、必要であれば LISS として上流のリポジトリに戻る。これは完成された静的な規律ではなく、エコシステムとして自己進化し続けるための器である。

ここでの「完成」とは、すべてのルールが揃った最終版のフィックスではなく、「ルールをどのように安全に更新するか」という構造の完成を意味する。ソフトウェアプロセスの文書は、一度書いたら終わりではない。AI ツールの仕様が変わる。エージェントが介入できる surface が増える。実運用でプロセスと現実の間に gap が見つかる。テンプレートが完成に近づくということは、変更が不可能になることではなく、変更の仕方がより明確に、より安全になるということだ。

## 未完成を隠さない勇気

`process-gap-register.md` は、高並列のエージェント運用や、複雑な障害復旧手順がまだ未成熟であると率直に自己申告している。もし真の完成形に「安全な並列運用」が含まれるのであれば、本テンプレートは今はまだそこに到達していない。ADR 0007 による worktree 分離は、その遠い目標に向けた技術的な布石にすぎない。プロセスの到達度を客観的な自己評価の対象とすることも、アーキテクチャ設計の重要な一部である。

未完成であることを包み隠さず明示することは、開発者への信頼の証である。AI 協働に関する言説は、しばしば「AIに任せれば何でもできる」という無責任な万能感に陥りがちだ。だが、実践的な運用テンプレートに必要なのは万能感ではなく、「現在の仕組みでどこまで安全に自動化でき、どこから人間の判断が不可欠で、どのプロセス gap が未解決なのか」を正確に言明できることだ。未完成の登録簿（Gap Register）は能力不足の告白ではなく、継続的改善のための起点である。

## 人間の承認点：Human-AI Interaction の実装

チェックポイントを設け、生成を委ね、レビューで止める——この構造は、Anthropic のエージェント設計解説（[Anthropic Research](https://www.anthropic.com/research/building-effective-agents)）が述べる agent workflow のベストプラクティスと同じ方向を向いている。しかし本テンプレートの独自性は、その承認点（Human-in-the-loop）を、ふんわりとした推奨ではなく、強固な文書と不可逆的なフェーズゲートとして物理的に固定した点にある。モデルが将来どれほど強力になっても、このゲートは残る。なぜならそれはAIの能力不足を補うためではなく、人間の責任と監査可能性をシステムに担保するための設計だからだ。

Bainbridge の自動化の皮肉（[ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/0005109883900468)）が鋭く指摘したように、自動化は人間を不要にするのではなく、しばしば人間に残る仕事を極端に難しくする。だからこそ、人間の役割を「最後に全部のコードを見て安全性を保証する人」にしてはならない。Adjudicator は、最も難しく、かつ価値のある設計判断が要求される位置に、あらかじめ戦略的に配置されるのである。

## 読み物と規範を混ぜない

この文章を含む `docs/research/` フォルダの内容は、人間がソフトウェア開発の哲学を読み解くための explanation（解説）である。AI エージェントがここを読むことを禁止するわけではないが、日々の作業の入力コンテキストとして読む必要は全くない。また、ここにある論考や references は、`llm-project-template` が提供する実務的なプロジェクト運用テンプレートのルールそのものに変更を加えるものではない。

この「規範」と「解説」の分離は、アーキテクチャ上極めて重要である。エージェントが従うべき絶対の運用規則は、`AGENTS.md`、ADR、`architecture/`、`collaboration/` などの文書に厳格に定義されている。`research/` は、その規則がなぜ必要なのかという背後の理由（Rationale）を厚く書く場所である。人間が思想を深く理解し、規則を形骸化させずに維持し、必要であれば議論を経て別の artifact へ昇格させるための肥沃な土壌であって、これ自体がエージェントへの直接的な命令書ではないのだ。
