# pull 型の上流：分散システムにおける主権と自己決定

2026-07-06。非規範。関連: ADR 0008、LISS-0001。

---

どのような優れたテンプレートであっても、プロジェクトにクローン（コピー）された瞬間から、それは否応なく古くなり陳腐化を始める。陳腐化はテンプレートの設計上の欠陥ではなく、時間が流れる以上避けられない構造的な帰結である。しかし、真の問題は「コードやルールが古くなること」そのものではない。上流（テンプレートのメンテナー）が下流（個別の採用先プロジェクト）のローカルな文脈を無視して、下流のコードを強制的に書き換えようとする（Push型の更新）ときに、悲劇は起こる。

だからこそ私は、強制的な Push ではなく、下流からの主体的な Pull（Pull型更新）をアーキテクチャとして採択する。いつ、どのタイミングで上流の変更に同期するかは、各プロジェクトの採用アダプター（採用先でテンプレートを運用する人・チーム。用語は [README](README.md)）が完全に決定する。上流は、世界中に散らばる下流プロジェクトのレジストリも、それらに書き込むためのアクセス権限（資格情報）も一切持たない（ADR 0008）。

テンプレートの配布と更新は、通常のソフトウェアライブラリ（NPM や PyPI パッケージ）の配布よりもはるかに厄介である。ライブラリであれば、更新は明確なインターフェース（API）の境界を越えてブラックボックスとして入ってくる。しかしテンプレートは、採用先のリポジトリのファイルシステム内へ直接コピーされ、そこで人間のエンジニアによって編集され、プロジェクト固有のドメインやアーキテクチャの「意味」を帯びて成長する。採用直後は全く同じファイルであっても、数週間後には下流の固有の設計判断が深く混入している。したがって、テンプレート更新における真の課題は「最新版のファイルをどうやって配るか」ではなく、「どこまでが上流が提供する普遍的な改善であり、どこからが下流が勝ち取ったプロジェクト固有の主権（Sovereignty）か」を安全にマージする問題なのである。

## 主権を既定で保護する：3-way merge の哲学

この複雑な課題に対し、本テンプレートは 3-way merge（3方向マージ）を採用する。テンプレート上流だけが変えた箇所は素直に更新し、展開先（下流）だけが独自に変えた箇所は絶対的に維持し、両方が同じ箇所を別々に変えた場合はコンフリクト（衝突）として人間に解決を求め、展開先が意図的に削除した箇所は、上流の更新時に無言で復活させない（ADR 0008 Decision 3）。naive（素朴）な単純上書きは、下流が丹念に築き上げたカスタマイズを一瞬で破壊する。この破壊のリスクは、Adjudicator 自身が ADR に重い代償として記録している。

特に「削除を尊重する」という挙動は、アーキテクチャ上極めて重要である。テンプレート提供側の観点では、削除されたファイルは単なる「欠落」や「エラー」に見えるかもしれない。しかし採用先の観点に立てば、その削除は「我々のプロジェクトにはこのルールは不要である」という明示的で高度な設計判断（Architectural Decision）である可能性が高い。上流がそれを自動的に復活させるならば、それはテンプレートが下流のローカルな設計判断を一方的に上書きする機構になってしまう。3-way merge は単なる git の技術的なマージアルゴリズムではなく、「各プロジェクトの主権と自己決定権を尊重する」という強力な哲学の表明なのだ。

[cruft](https://github.com/cruft/cruft) や [Copier](https://copier.readthedocs.io/en/stable/updating/) は、まさにこの "boilerplate drift"（テンプレートと実プロジェクトの乖離）という難問に長年取り組んできた偉大なツール群である。本テンプレートはそれらの哲学に深く共鳴しつつも、追加のツール依存（Python パッケージなど）をプロジェクトに強要しないよう、純粋な `git merge-file` ベースで同型の構造を実装することを選んだ。

## 同期も通常の変更である：Docs-as-Code の徹底

update script によるテンプレートの更新は、プロジェクトの trunk（mainブランチ）へ直接コミットしてはならない。必ず専用のブランチを作成し、Pull Request（PR）を発行し、通常の CI を通過しなければならない（ADR 0008 Decision 4、ADR 0007）。「上流の公式テンプレート由来だから無条件で安全だ」というような特別扱いは一切しない。[Renovate](https://docs.renovatebot.com/) や Dependabot が業界に普及させた「外部からの更新は自動で PR として提案され、人間のレビューと CI のテストをすべて通って初めて取り込まれる」という堅牢なパターンの、テンプレート版である。

ここで注意すべきは、"A clean merge is not the same guarantee as 'nothing needs review'"（クリーンなマージができたからといって、レビューが不要だという保証にはならない）という点だ（ADR 0008 Consequences）。機械的にコンフリクトなくマージできたこと（構文的成功）と、その変更がプロジェクトの文脈において正しいこと（意味的正しさ）は、全く別次元の話である。

PR は単なる消化試合の運用手続きではなく、「意味論的レビュー（Semantic Review）」を行う場である。テンプレートの更新は、一見すると些細な Markdown の文言差分に見えても、エージェントの挙動（agent behavior）、フェーズの承認ゲート、プライバシー予算、あるいはブランチの規律に重大な影響を与えるかもしれない。CI は Git の conflict marker の残りや、Markdown の形式崩れを機械的に見つけることができる。しかし、その変更が「我々の採用先の文化や、過去に積み上げた既存の ADR と矛盾しないか」を判断できるのは、ローカルの文脈を理解している人間の Adjudicator だけである。

## 改善は下流から還流する：エコシステムの循環

採用アダプターによる実運用の泥臭いフィードバックは、LISS-0002 や LISS-0005 といった Issue として上流へと戻る（還流する）。[GitLab Flow の upstream first](https://about.gitlab.com/topics/version-control/what-are-gitlab-flow-best-practices/) は、バグ修正や共通の改善をまず上流（main）へ組み込むという大原則を説いている。本テンプレートの「現場のフィードバックを LISS 化して抽象化し、そこから世界中へ再配布する」という運用は、この Upstream-First のアーキテクチャと同型である。


## 権限委譲と自己決定の系譜：中央統制の限界

この「上流は強制せず、下流が主権を持ってPullする」という設計は、単なるGit運用の技術論ではなく、ソフトウェア工学に絶大な影響を与えた「分権化」の歴史的系譜に深く連なっている。特に、以下の2つの歴史的ブレイクスルーが本テンプレートの思想的基盤（エビデンス）となっている。

第一の基盤は、製造業における「PushからPullへの転換」である。かつて主流だったフォード生産方式のような中央集権的な大量「Push」型の管理は、末端の状況変化に対応できず膨大な無駄を生んだ。これに対し、大野耐一が確立したトヨタの「かんばん方式（Kanban）」は、上流がスケジュールを押し付けるのではなく、下流（後工程）が必要な時に必要な分だけ引き出す（Pull）仕組みへと転換した。この中央統制を捨て現場の適応力を最大化する思想は、後にリーンソフトウェア開発（Lean Software Development）の土台となり、ソフトウェア産業に多大な影響を与えた（Poppendieck & Poppendieck, 2003）。

第二の基盤は、現代組織における「Empowered Execution（権限委譲された実行）」の概念である。スタンリー・マクリスタル将軍の著書『Team of Teams』で詳述されたように、現代の米軍は、従来の縦割りのヒエラルキー（司令部からのCommand and Control）では、分散ネットワーク型で変化の速い対象に対応しきれなくなった。その解決策は、司令部が「共有された意識（Shared Consciousness）」を提供しつつ、実行の意思決定を現場のチームへ完全に委譲することであった。中央からの指示を待つのではなく、現場の文脈を知る最前線のチームが自律的に意思決定を下すこのネットワーク型の構造は、今日のアジャイル開発が目指す自律的なスクラムチームの姿と本質的に同型である。

本テンプレートが「アーキテクチャの指針（共有された意識）」だけを公開し、それをいつ、どのように取り込むかの「自己決定権（Sovereignty）」を採用先プロジェクト（現場のチーム）に完全に委譲しているのは、単なる手抜きではない。製造業と組織論という異なる領域の実践が繰り返し示してきた「変化が激しい環境下では、中央統制よりも現場への権限委譲と Pull 型の情報取得のほうがスケールする」という教訓に沿った、意図的なアーキテクチャ上の決定である。もっとも、これらは回顧録や実践報告に属する証拠であって、統制された実証研究ではない。だからこそ本テンプレートはこの教訓を無条件の真理としてではなく、ADR 0008 という自プロジェクトの文脈における決定として採用している（[エビデンスと規則](2026-07-06-rationale-evidence-based-process-design.md)）。

## 上流は下流を知らない：非中央集権の規律

この Pull 型の設計は、スケーラビリティとセキュリティ（信頼）を同時に手に入れる。上流は下流を管理しないため、採用先のプロジェクト数が10であろうと10万であろうと、上流の運用コストは一切増加しない。また、上流に各社の GitHub トークンなどの資格情報を集中させる必要がないため、サプライチェーン攻撃の巨大な単一障害点になることもない。

この絶大なメリットの対価として、上流は「同期（Pull）をサボるプロジェクトは、古いまま永遠に放置される」という事実を受け入れる（ADR 0008 Negative）。これは怠慢ではなく、強制よりもプロジェクトの主権を優先した、一貫して意図的な選択である。中央集権的な統制と品質保証を潔く諦めること。上流はすべての採用先が常に最新であることを決して保証しない。採用先は、自分たちのビジネスのタイミングで主体的に pull し、自分たちの責任によるレビューで取り込む。テンプレートが世界に提供するのは、支配（Control）ではなく、持続可能な更新可能性（Updatability）なのである。

`research` フォルダを同期の対象から外すという判断（[規範と読み物](2026-07-06-rationale-normative-vs-reading-documents.md)）も、この主権方針の一貫した現れである。下流に配るべきはプロジェクトを縛る「規範」だけであり、上流メンテナーの哲学やエッセイである「読み物」は上流のローカルに留める。読み物をどれほど充実させても、世界に配布される運用テンプレート（ルールそのもの）は変わらない。規範自体を変える必要があるなら、別途 ADR を書き換え、Pull 型の更新経路のネットワークに乗せなければならない。
