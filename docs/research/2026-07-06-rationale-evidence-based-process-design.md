# エビデンスと規則：EBSEとコンテキストの尊重

2026-07-06。非規範。関連: LISS-0002、LISS-0005、`process-gap-register.md`。

---

ソフトウェア開発プロセスの「規則」は、往々にして業界の流行、声の大きいテック企業の事例、あるいは一部のシニアエンジニアの極端な成功体験によって恣意的に決まりがちである。「そのベストプラクティスは、自分たちの現在のチーム規模、プロダクトのフェーズ、技術的負債の状況といった『固有の文脈』において本当に効果を発揮するのか？」という問いを冷静に確かめる習慣は、この業界にまだ十分に定着しているとは言い難い。私は、規則を硬直化させる前に調べ、自プロジェクトの文脈と突き合わせ、人間の Adjudicator が責任を持って決定するという作法を採る。

AI との協働プロセス設計においては、この流行に飛びつく誘惑がとくに強い。新しいエージェントの機能、新しいプロンプトエンジニアリングの技法、新しい「AI-Native なベストプラクティス」が毎週のように現れる。話題性は極めて高いが、それが自分たちの厳密な実務文脈においてどのような副作用をもたらすのかは誰にも分からない。客観的な証拠なしにそれらを規範（ルール）へと昇格させれば、テンプレートはあっという間に流行の堆積物、あるいは矛盾したルールのフランケンシュタインになる。だからこそ、私は「調査」と「規範」を厳格に分離する。調べたことや議論の過程は `research` として残し、厳しい評価を生き残り、プロジェクトで実際に採用されたものだけを「規範（ADR）」へと上げる。

「先行研究や実証研究、先行サービスの事例、ベストプラクティスなどを調べて、分析して、相談に乗って。」

これは、`develop` ブランチの扱いを直観や気分で決めない、という現場の切実な要求として出た言葉だ。しかしこれは単発の要求ではなく、一般則として三つの堅牢な行動パターンに昇華される。第一に、規則を定める前に調査する。第二に、エージェントや採用アダプターからの失敗フィードバックを改訂の一次入力にする。第三に、調査と規範を分離して両方の記録を残す。

## EBSE (Evidence-Based Software Engineering) の系譜

Kitchenham, Dybå, Jørgensen の記念碑的論文（[ICSE 2004 PDF](https://web-backend.simula.no/sites/default/files/publications/SE.5.Kitchenham.2004.pdf)）は、医学界で革命を起こした「根拠に基づく医療（Evidence-Based Medicine）」の手法を、ソフトウェア工学へ移植すること（EBSE）を提唱した。続く Dybå et al. の研究（[IEEE Software 2005 PDF](https://web-backend.simula.no/sites/default/files/publications/Dyba.2005.1.pdf)）は、客観的エビデンスに基づかない思い込みでの技術採用が、プロジェクトに致命的な誤判断を生むと警告している。

EBSE は医学を範に取るが、そのまま無邪気に輸入することはできない。医学には二重盲検による無作為化比較試験（RCT）の伝統があるが、ソフトウェア工学における現場実験は、人間のスキルや組織文化という文脈依存（Context-Dependency）があまりにも大きく、厳密な再現が不可能に近いからだ。だからこそ EBSE が説くのは、「客観的証拠さえ見つかれば自動的に決まる」という機械論ではなく、「証拠を批判的に評価し、自分たちの特定の文脈（Context）へと統合する」という高度に知的な手続きである。本テンプレートの「調査 + 文脈 + Adjudicator 判断」というフローは、この EBSE の手続きを小規模な日々の運用へと写し取ったものである。

EBSE の基本手順——(1) 問いの定式化、(2) エビデンス探索、(3) 批判的評価、(4) 実務文脈への統合——は、先述した `develop` 検討で私たちが踏んだ手順と完全に同型である。問い（長命ブランチを全面禁止すべきか）、DORA や GitFlow 原著者・各メガテック企業のプラクティス調査、「存在そのものではなく統合遅延こそが問題」という批判的評価、そして統合契約付き許可というプロジェクト固有の規則案の提示。

## データで直観を修正する：DORAの教訓

[DORA](https://dora.dev/capabilities/trunk-based-development/) は、長年の大規模調査により、プロセスの実践とデリバリー性能の相関を統計的に測定している。「develop ブランチは悪であるから全面禁止すべきだ」という初期の単純な直観は、DORA のデータの操作的定義——「統合頻度」——を読み解くことで、より精緻な見解へと修正できた。規則の背後に実証研究を置くという要求が、実際に感情論を排除し機能した好例だ。

Forsgren, Humble, Kim の *Accelerate*（[IT Revolution](https://itrevolution.com/product/accelerate/)）は、DORA の調査手法——大規模サーベイに基づく統計的分析で技術・組織の実践とシステム性能の関係を測る——をまとめたものである。私たちが学ぶべきは、彼らの個々の主張に盲目的に同意することではない。「個人の直観や声の大きさではなく、客観的なデータで実践を評価する」という態度そのものである。

ただし、その態度は常に次の強い注意とセットである。「相関関係は文脈に依存する。他所のメガテック企業でのエリート実践が、自分たちの予算や制約に自動的に当てはまるわけではない。」

## フィードバックが規範を動かす：プロセスギャップ

「採用アダプターからの報告 → ギャップ分析 → 規範改訂」。本テンプレートの `process-gap-register.md` は、理想と現実の間にギャップを見つけたら、黙ってやり過ごすのではなく文書を書いて登録簿に記録せよと厳格に定める。LISS-0002（第1回）、LISS-0005（第2回）がその実績である。システムはトップダウンの規則だけでなく、現場からのボトムアップの観測によって自己修復していく。

## ISSUE と ADR が要求する「証拠」と「文脈」

本テンプレートが ISSUE（計画）や ADR（アーキテクチャ決定記録）において、エージェントに対して一貫して「根拠となる資料（Reference）」と「プロジェクトの文脈（Context）」の提示を強く求めるのは、まさにこの EBSE の哲学をプロセスとして強制するためである。

「なぜこの設計にするのか」「なぜこのライブラリを入れるのか」という問いに対し、単なるAIの推測や「一般的に普及しているから」という流暢な散文は許容されない。ADRのフォーマットに「Context（文脈）」と「Consequences（影響）」というセクションが設けられているのは、Michael Nygard の原典が示す通り、その決定がどのようなローカルの制約のもとで、どのような証拠に基づいて下されたかを記録するためだ。

AI 協働において、この制約は二重の意味を持つ。エージェントは学習データ内の無数の「一般的なベストプラクティス」を瞬時に引き出せるが、それが自プロジェクトの文脈（予算、フェーズ、既存システムの制約）に合致するかどうかは評価できない。したがって、ISSUE で作業を計画し ADR を起草する際、エージェントには「採用する技術の公式ドキュメントや先行事例へのリンク（客観的証拠）」と「なぜそれが今の私たちの状況に必要なのか（文脈への統合）」をセットで言語化させ、人間の Adjudicator による検証ゲートを通す。これが、テンプレートが要求する Evidence Discipline（証拠の規律）の実体である。

## 外部ソフトウェアと依存関係の証拠に基づく採用

EBSEの文脈への統合という思想が最も強烈に現れるのが、外部リソース（ライブラリ、外部API、フレームワークなど）の採用プロセスである。AIエージェントは一般的な知識から「この機能なら〇〇というライブラリが標準的です」と無邪気に提案してくる。しかし、それが私たちのチームのセキュリティ要件、依存関係の競合、保守の体制（Context）に適合するかどうかは、AIには判断できない。

そのため本テンプレートは `docs/architecture/external-resource-adoption-contract.md` という運用規約を設け、外部由来の **resource**（AI 生成物や持ち込みデータ等。ソフトウェア依存そのものは `dependency-policy.md`）を信頼境界へ入れる前に、厳格な採用ライフサイクルを要求する。検査判定（`accepted | rejected | needs_recheck`）と、信頼利用への投入（`accepted -> adopted`）は別であり、accepted だけで adopted とみなさない（用語は [README](README.md)）。「なんとなく便利だから」ではなく、「このプロジェクトの文脈において安全で適切である」という証拠（Evidence）を揃え、検証し、Adjudicatorが明示的に採用を承認する。これもまた、EBSE の実践的な防衛線である。

## 調査したから正しい、ではない：限界への自覚

ソフトウェア工学のエビデンスは、医学や物理学のエビデンスほど強固ではない。RCT は稀であり、文脈依存が圧倒的に大きい。だからこそ「調査結果 + 自プロジェクトの文脈 + Adjudicator 判断」の三点で決定を下す。research は非規範の「考察」のまま残し、規範化は Adjudicator の承認後にのみ行う——これは EBSE の「実務文脈へ統合する」ステップの実装として極めて妥当だ。

この慎重さは、AI が生成する「もっともらしい根拠（ハルシネーション）」にも向けられる。エージェントは説得的な理由をいくらでも大量に書ける。だが、説得力は決して証拠ではない。ドキュメントが外部の出典 URL と取得日を厳格に要求し、未検証のものを明記するのは、AI 時代における最小限の防衛線だ。出典のない主張は、どれほど流暢でも、規範へは昇格させない。技術のエビデンスは恐ろしい速度で陳腐化するため、取得日を明記して将来の再検証に備える。

最後に、この読み物自体も規範ではない。ここで述べた EBSE の姿勢を実際の運用へ組み込むかどうかは、ADR や collaboration 文書が担う仕事であり、`docs/research/` の哲学的編集はテンプレートの運用規則を強制的に変更するものではない。証拠を集め思索することと、規則を変え行動することを混同しない——それ自体が、本稿が主張する EBSE の姿勢の実践である。
