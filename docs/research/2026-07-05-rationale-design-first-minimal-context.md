# 設計先行とコンテキスト境界のコントロール

2026-07-05。非規範。関連: ADR 0001。

---

LLMを操作する際、「賢いプロンプト」を競う文化は、ソフトウェアエンジニアリングの本質を見落としている。真の課題は、言い回しの巧みさではなく、モジュール設計と同様の「境界のコントロール」にある。何を渡すか、何を渡さないか、そしてどの操作をどの道具に任せるか。自律的に動くコーディングエージェントを誘導し制御する最も確実な手段は payload（入力コンテキスト）の設計であり、include/omit（含める/外す）の選択は、サンドボックスへ何を見せ、何を推測するなと伝えるアーキテクチャ統合層の核心である（[エージェントはサンドボックスである](2026-07-07-rationale-saas-agent-as-sandbox.md)）。

私は、コンテキストの選択を「個人のプロンプトエンジニアリング技法」から、「レビュー可能な設計成果物」へと格上げする。すべての依頼は設計ステップから始まり、テストと実装はその後に従う。

設計インテーク（Design Intake）では、対象となる挙動、VO/DTOの候補、PortsとAdaptersの境界、含める文脈と外す文脈、モデルやツールの振り分け、そして「推測してはならない境界」を先行して決定する（ADR 0001）。設計は単なる儀式ではない。意思決定の記録である。優れた設計インテークは、エージェントに対して「何をしてよいか」を指示するだけでなく、「何を知らないまま進めてはいけないか」というガードレールを提供する。

## コンテキストは多いほどよい、という幻想

より小さいペイロードで目的が達成できるなら、リポジトリ全体を無批判に送ってはならない。その理由はトークンコストや処理時間だけではない。無関係な文脈は、モデルの推論性能を物理的に劣化させるからだ。

Shi et al. の研究（[arXiv:2302.00093](https://arxiv.org/abs/2302.00093)）は、無関係な情報を問題文に加えるだけで LLM の推論精度が有意に低下することを示した。payload は厳密に選んで送る。これはベストプラクティスというより、実証に基づく制約に近い。

また、Liu et al. の "Lost in the Middle"（[arXiv:2307.03172](https://arxiv.org/abs/2307.03172)）は、非常に長いコンテキストを扱えるモデルであっても、関連情報が文書のどこに位置するかによって情報の利用性能がＵ字型に変わる（最初と最後は思い出しやすいが、中盤は忘れやすい）ことを示した。大きなコンテキストウィンドウは万能の記憶領域ではない。むしろ、長い窓を持つほど「何を入れるか」の設計責任が人間の側に重くのしかかる。大量の文書を渡して「うまく読め」と指示するのは、設計の放棄である。

コンテキストは希少資源として扱うべきだ。Anthropic の context engineering 解説（[Anthropic Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）は、agent に渡す情報を "informative, yet tight"（情報豊富でありながら、引き締まった状態）に保つことを勧める。include/omit の台帳を設計の必須項目にしているのは、この tight なコンテキストを実現するための具体的な実装形態である。

## omit は消極的な省略ではない：推測の境界線

ソフトウェア設計において、多くの文書は「システムが何をするか（何を含めるか）」を書く。しかし、AIとの協働においては、「システムが何をしないか（何を含めないか）」も同じくらい重要である。omit（除外）は単なる節約ではない。モデルに対して推測の境界を引く、積極的な設計行為である。

例えば、未決のデータストア選定を含む議論の過程を payload に入れてしまうと、エージェントはその語彙を見て既成事実のように扱い、実装に混入させるかもしれない。関係ない過去の実装例を渡すと、現在の新しい設計方針と異なる古いパターンを模倣するかもしれない。プライベートなデータを広く渡すと、必要以上のデータ露出となりセキュリティリスクを生む。omit は、これらの危険を「注意してね」という曖昧な自然言語による依頼ではなく、入力データの物理的な遮断として閉じ込める。

`docs/collaboration/privacy-context-budget-policy.md` が payload budget（コンテキスト予算）を置くのはこのためである。プライバシー保護と推論性能改善は相反するものではなく、同じコインの裏表である。不要なものを渡さないことは、情報漏洩の表面積を減らし、推論のノイズを減らし、人間がレビューすべき根拠の範囲を絞り込む。

## 能力の階段：責任分界とコスト最適化

機械的なフォーマット作業は決定的ツールへ。単純な補完は小さいモデルへ。境界判断、曖昧なドメインロジックの解釈、プライバシーが絡む高度な判断は強い推論モデルへ。最下段の原則は極めて単純だ。「決定的ツールで済むことに、確率的な LLM を使うな」。

モデル振り分けの経済性は Chen, Zaharia, Zou による FrugalGPT の研究（[arXiv:2305.05176](https://arxiv.org/abs/2305.05176)）が裏づけ、その後の RouteLLM（[arXiv:2406.18665](https://arxiv.org/abs/2406.18665)）は、問い合わせの難度に応じた強弱モデルの振り分け自体を学習可能な問題として定式化した。安価なモデルで十分なタスクに強力なモデルを呼ばないことは、単なるクラウド費用の節約にとどまらない。強いモデルを乱用すると、判断の所在が曖昧になるというアーキテクチャ上のリスクが生じる。

コードフォーマッターや Linter が決定すべきことを AI に聞けば、出力の再現性が下がり、CI（継続的インテグレーション）が不安定になる。依存境界の違反チェックを人間の説得的文章で済ませてしまえば、機械的な制約としてコードベースに残らない。「能力の階段」とは、コスト最適化の手段である前に、責任分界点の設計なのだ。

ここでいう「強い推論」も、万能の権限ではない。Architecture Path で強い推論エージェントを呼ぶのは、ADR（アーキテクチャ決定記録）を勝手に採択させるためではない。人間のアーキテクト（Referee）が判断するための選択肢と根拠を整えさせるためである。道具の能力が上がるほど、道具が自律的に決めてよいことと、人間が最終的に決めることの境界線をより明示的に設計する必要がある。

## 未決は空白ではない：未知の管理

要件定義において、未決定の事項は「何もない空白」ではない。それは「まだ決まっていない」という明確な状態である。`Current Non-Decisions` に載る事項は、エージェントが推測で勝手に埋めてよい領域ではない。それは将来の ADR のための話題である。"Must not guess" は設計ノートの必須欄だ。入力の曖昧性を制御する手法としては、要求工学における EARS（[IEEE Xplore](https://ieeexplore.ieee.org/document/5328509)）などの構造化アプローチも参考になる。

曖昧な要求をそのまま大きな payload と一緒に渡すと、モデルは空白を「尤もらしい幻覚」で埋めようとする。これはモデルの欠陥というより、次世代のトークンを予測し続ける生成モデルの本質的な性質である。だからこそ、空白を「空白」としてシステムに認識させ、それ以上進まないようにする構文が必要になる。設計ノートの `Ambiguities`、`Open decisions`、`Stop conditions` といった項目は、単に作業を止めるためだけでなく、推測で埋めてはいけない領域を明示的に保存するための防壁である。

## 設計先行はウォーターフォールではない

「設計から始める」と言うと、古典的なウォーターフォール開発における重厚長大な事前設計（Big Design Up Front）を連想するかもしれない。しかし、本プロジェクトにおける設計インテークは、すべてを事前に決めるためのものではない。むしろ、「今決めるべきこと」と「今はまだ決めてはいけないこと」を分けるための、極めてアジャイルで軽量なゲートである。

Fast Path における compact design note は、その証拠である。タイポの修正や機械的なリファクタリングに大げさな設計書は要らない。それでも、scope、omitted context、deterministic checks は数行で短く述べる。設計先行の価値は、文書の量ではなく、作業前に「変更の境界」を意識することにある。

Feature Path では、受入仕様と phase gate が設計の中心になる。Architecture Path では、規則を変えるのか、読み物を書くのか、ADR を発行するのかを区別する。今回の `docs/research/` 編集のような作業では、この区別が特に重要である。読み物は哲学的に厚く書いてよい。しかし運用テンプレートの根幹規則は勝手に変えない。この境界を最初に設定するからこそ、特定の領域内では自由で創造的な作業が可能になるのだ。

## 不確実性を閉じ込める：設計の規律

決定的ツールを優先する原則は、LLM の持つ本質的な不確実性を意図的に狭い場所へ閉じ込める設計である。システム内で不確実性を扱う場面を最小限に減らすこと自体が、AI時代のソフトウェアエンジニアリングの規律の一部だ。これは [エビデンスと規則](2026-07-06-rationale-evidence-based-process-design.md) と対をなす考え方である。

設計先行のアプローチは、AI の作業スピードを遅くするための無駄な儀式ではない。AI の圧倒的な生成速度を、プロジェクトが安全に受け止められる形（消化可能な変更）へと変換するための構造である。速く生成できることが、そのまま速く統合できることを意味するわけではない。継続的に統合可能なコードを生成するためには、「何を根拠に」「どの境界で」「どのフェーズまでを進めるのか」を、手を動かす前に決める必要がある。
